# Must Solve 設計問題清單
> 狀態：Planning 階段核心設計缺口追蹤
> 最後更新：2026-06-22
> 用途：把「目前最欠的 7 個遊戲設計架構問題」集中管理；每解決一項，就在本檔打勾，並視影響同步補進 `game-design-plan.md`、`game-architecture-plan.md`、`planning-dashboard.md` 或專用主檔。

---

## 使用規則

1. 每次只集中解一個 Must Solve 問題。
2. 每次討論都先說「目前問題」，再說「我的建議」，再判斷「能不能用模擬解決」。
3. 如果能用模擬解決，先用模擬找出合理範圍，再進入正式設計結論。
4. 解決後，把該項從 `[ ]` 改成 `[x]`。
5. 視影響同步更新核心文件：
   - `game-design-plan.md`：補玩法規則、數值、流程或 UI 影響。
   - `game-architecture-plan.md`：補資料結構、同步邏輯、MVP 範圍或技術風險。
   - `planning-dashboard.md`：更新狀態與下一步順序。
   - 專用主檔：波次看 `waveplan.md`，卡片看 `bosscard.md`，MVP 驗證看 `mvp-validation.md`。
6. 如果某項只完成部分決策，先保持 `[ ]`，並在該項下方加「已確認」與「仍待確認」。

## 每次討論固定格式

之後每次討論 `mustsolve.md` 的任一項時，固定用下列格式開場：

```md
## 目前問題

我們現在要解的是：Must Solve X：XXX

這個問題卡住的是：
- A
- B
- C

如果不解決，會造成：
- 玩家體驗問題
- 數值平衡問題
- 開發實作問題

## 我的建議

我建議先採用：XXX

原因：
- 它最適合 MVP
- 它最容易測試
- 它不會太早把系統做死

## 能不能用模擬解決？

可以 / 不適合 / 部分可以。

如果可以模擬：
- 要模擬什麼
- 需要哪些輸入數值
- 輸出結果看什麼
- 什麼結果算合理
```

### 模擬適用規則

適合用模擬解決的項目：

- 怪物血量、攻擊力、速度。
- 1-30 關難度曲線。
- 方塊產出量與建築成長速度。
- 核心 DPS 是否打得過某波怪。
- 銀幣、票券、抽獎、裝備成長速度。
- 多人效率差距。
- 卡片強度是否過強或太弱。

不太適合只靠模擬解決的項目：

- 玩家覺得好不好玩。
- 操作手感。
- UI 是否清楚。
- 合作時玩家會不會自然溝通。
- 建築玩法是否有創意感。

這些項目要靠 MVP 實玩驗證。模擬只能先排除明顯不合理的數值。

### 每個 Must Solve 的處理方式

| 項目 | 模擬適用度 | 處理方式 |
|---|---|---|
| Must Solve 1：核心樂趣循環 | 部分可以 | 主要靠設計討論 + MVP 實玩驗證；模擬輔助檢查玩家每分鐘可挖多少、可蓋多少、晚上有沒有壓力 |
| Must Solve 2：怪物職能與敵人設計 | 非常適合 | 模擬每關敵人血量、核心傷害、清怪時間、核心受傷量 |
| Must Solve 3：建築策略與方塊取捨 | 部分可以 | 模擬寬地基、窄高塔、厚防線、攻速塔的生存差異；好不好玩仍要實測 |
| Must Solve 4：難度曲線與 1-30 關節奏 | 非常適合 | 用模擬決定 1-30 關初版曲線 |
| Must Solve 5：卡片系統與局內流派 | 部分可以 | 數值卡可以模擬；玩法型卡片需要設計判斷 + 實玩 |
| Must Solve 6：多人合作分工 | 部分可以 | 模擬 1-4 人資源效率和怪物壓力；合作樂趣需要實測 |
| Must Solve 7：MVP 驗證標準 | 不主要靠模擬 | 定義做出來後要怎麼測，包含模擬測試與真人實玩測試 |

### 預設原則

- 每次只解一個 Must Solve，不混著解。
- 能模擬的項目，先模擬再定數值。
- 模擬結果只是初版設計依據，不是最終答案。
- 最終是否打勾，必須同時滿足：問題有結論、文件已補上、後續不用再猜。

## Must Solve 1：核心樂趣循環

- [x] 明確定義玩家每 10 秒、每 1 分鐘、每 10 關的有趣決策。

### 需要補完

- 白天挖礦時，玩家在「安全資源」與「高價值資源」之間的取捨。
- 晚上防守時，玩家除了等待核心攻擊外，還能做什麼有效操作。
- 王關前後，玩家如何根據局勢調整建築與卡片選擇。
- 多人模式中，玩家是否會自然形成挖礦、搬運、修復、守核心等分工。

### 解決標準

- 可以用一段短流程描述玩家從第 1 關到第 10 關的主要體驗。
- 每個階段至少有一個明確的風險/回報選擇。
- MVP 可以驗證這個循環是否好玩。

### 已確認

- 白天與晚上都可以挖礦、建塔、修復。
- 白天偏低風險規劃；晚上偏高風險判斷。
- 晚上離家時，如果核心扣血，需要出現核心危險警告。
- 修復核心每秒消耗 1 疲勞值。
- 修復量 = 修復能力 / 60；基礎修復能力 50 → 每秒回復 0.83 血。
- 疲勞值每個白天/晚上依照疲勞能力回復；基礎疲勞能力 60 → 每階段回復 60，上限固定 120。
- Must Solve 1 的早期循環驗證用怪物攻擊力暫定為：平民 1、猛男 2。
- 完整怪物血量、攻擊力、速度與 1-30 關配置表留到 Must Solve 2。

### 模擬審計結果

- 礦山距核心假設 20 格、長按挖礦 5 下/秒、基礎挖掘力 10、背包 50 重量時：
  - 左礦山 A 約 11.4 塊/分鐘。
  - 右礦山 B 約 11.4 塊/分鐘。
  - 背包平均約 4.5-4.8 塊會滿，約每 20-25 秒需要回核心一次。
- 基礎修復能力：
  - 60 疲勞約可回復 50 血。
  - 120 疲勞約可回復 100 血。
- 攻擊力壓力測試：
  - 平民攻擊力 1 時，早期有晚上離家挖礦的空間。
  - 平民攻擊力 2 時，第 1 關可過但偏危險，第 3 關開始容易壓力過高。
  - 平民攻擊力 3 時，第 1 關就可能死亡，不適合早期。

### 合格標準

- 玩家白天可以在挖礦、建塔、修復之間切換。
- 玩家晚上也可以做同樣的事，但離家挖礦有核心失守風險。
- 核心扣血時有危險警告，讓玩家有機會回家救援。
- 修復不是免費拖時間，而是消耗疲勞值。
- 早期至少存在「晚上出去挖 10-30 秒」的可行空間。
- 如果玩家完全不建塔、不修復，核心會逐漸撐不住。
- 如果玩家有判斷地建塔/修復/挖礦，應該能感覺自己是靠選擇活下來。

### 已同步文件

- `game-design-plan.md`：已補核心樂趣循環、修復疲勞、早期怪物攻擊力暫定值。
- `game-architecture-plan.md`：已補 MVP 驗證與模擬審計指標。
- `planning-dashboard.md`：已更新 Must Solve 1 狀態。

## Must Solve 2：怪物職能與敵人設計

- [ ] 補出怪物類型、職能、血量、攻擊力、速度與出現節奏。（半成品：MVP 可先用目前敵人表與 `waveplan.md` 開工；完整職能手感需實作 / 實玩驗證）

- 怪物要能逼玩家改變建築策略，而不是只堆更多數值。

### 解決標準

- 第 1-30 關有可讀的敵人配置表。
- 每種敵人都有清楚用途：測試血量、測試攻速、測試射程、測試防禦、測試建築形狀等。
- 怪物血量、攻擊力、速度有初版數值。

### 已確認（單人 1-15 粗測）

- 已建立前 15 關候選敵人表，包含平民、跑者、猛男、盾兵、工兵、小隊長。
- 第 10 關小隊長暫定為第一個裝備門檻：血量 260、攻擊力 3、無特殊技能。
- 單人裸裝測試目標成立：裸裝可摸到第 9 關，裸裝第 10 關不能過。
- 抽數門檻粗測成立：20 抽未必能過第 10 關，50-100 抽開始能過第 10 關。
- 第 10 關後卡片基準：
  - 資源型基準：2000 挖掘量數。
  - 金輪核心：核心攻速 +0.20，作為第 10 關後強卡基準。（原 +0.25 在 Must Solve 4 多人正式加時測試中偏強）
  - 厚土外殼：血量上限 +25，偏普通防守卡。
  - 鐵牙核心：攻擊力 +2，偏普通攻擊卡。

## Must Solve 3：建築策略與方塊取捨

- [ ] 補強建築形狀、泥土地基、第二層方塊、梯子與多層建築的策略價值。（部分確認 / 待格子模擬）

### 已確認

- 只有與核心連續相連的泥土格，才算核心地基；孤立泥土不生效。
- 放置與拆除都必須維持連通性（禁止放孤立泥土、禁止拆導致斷連的泥土）。
- 核心攻擊覆蓋 = 所有連接核心的泥土格各自產生覆蓋後合併，非單一圓形。
- 所有連接核心的泥土格都是核心 Hitbox，核心只有一條血。
- 第二層方塊不是防禦牆，怪物攻擊到任何核心地基（包含泥土上的第二層）都算核心扣血。
- 怪物攻擊高度 = 怪物身高 + 攻擊距離（平民身高 2 格，特殊單位身高 3 格；後續格子模擬後可再調整）。
- 修復觸發條件：玩家站在核心地基（任何連接核心的泥土格）上 + 按鍵（桌面 R / 手機 🔧）。
- 初版 3 種建築策略方向已確認（集中地基 / 延伸地基 / 高層地基），詳見 game-design-plan.md。

### 仍待確認

- 格子化模擬：不同建築形狀（寬地基 vs 延伸線 vs 高塔）對生存率的實際影響。
- 梯子的實戰策略價值（目前只確認功能，未驗證是否有明顯策略意義）。
- 遠程怪物出現後，高層地基策略是否真的被克制（需要怪物設計確定後才能模擬）。
- MVP 實玩驗證：泥土 Hitbox 風險/回報機制是否真的有趣。

### 需要補完

- 連通泥土會提供攻擊覆蓋與核心 Hitbox；實際風險/回報需用格子化 hitbox、怪物高度、攻擊距離驗證，不能再使用「外擴就一定更早受擊」的舊假設。
- 高度、多層建築、梯子是否有實戰價值。
- 方塊不應只等於線性數值加成，也應影響戰場形狀。

### 解決標準

- 玩家能從不同建築流派中做選擇，例如寬地基、窄高塔、厚防線、攻速塔等。
- 至少定義 3 種早期可行建築策略。（✅ 初版方向已定，數值待格子模擬驗證）
- MVP 能測試泥土 Hitbox 機制是否真的有趣。

### 模擬方向（後續格子化模擬）

```
1. 找出所有與核心連通的泥土格
2. 每個泥土格各自產生攻擊覆蓋
3. 合併所有覆蓋區，形成核心攻擊區
4. 怪物路徑若進入攻擊區，核心可以攻擊
5. 怪物攻擊距離內若有任一核心 Hitbox，核心扣血
```

## Must Solve 4：難度曲線與 1-30 關節奏

- [x] 補出第 1-30 關的難度曲線與 30 關後 Endless 成長方向。（MVP 定案：1-20 為主要校準區，21-30 暫作阻擋區，後續進入 Must Solve 5）

### 需要補完

- 怪物血量成長曲線。
- 怪物攻擊力成長曲線。
- 怪物數量與種類成長曲線。
- 玩家單局方塊成長、長期裝備成長、卡片成長之間的對照。

### 解決標準

- 有一張第 1-30 關的節奏表。
- 第 10、20、30 關的王關定位清楚。
- 30 關後 Endless 不是只寫「爆炸式提升」，而是有初版方向。

### 歷史測試摘要

> 下列保留 Must Solve 4 的推導過程，方便回查為什麼這樣定案。開工時不要逐條引用這裡的舊輪次數值；正式規則以 `Docs/waveplan.md` 為準。

- 早期 15-30 關曾以 2 人房裸裝作為初版基準。
- 測試條件：
  - 2 名玩家。
  - 無裝備、無技能點、無卡片。
  - 白天兩人都挖礦建塔。
  - 晚上兩人都守家修復。
  - 多人不套用單人固定 +15% 靈動加成。
  - 兩名裸裝玩家各提供基礎靈動 50 → 合計約 +10% 核心攻擊/攻速。
- 第一輪結果：
  - 可通過第 18 關。
  - 第 19 關死亡。
  - 16-18 壓力逐步上升合理。
  - 第 19 關跳幅過硬，需要調整 19-20 關曲線。
- 第二輪結果：
  - 第 19 關只把工兵從 2 隻降回 1 隻時，2 人裸裝可極限通過，約剩 20 血。
  - 第 20 關 Boss 仍然明顯打不過，適合作為第二個裝備/卡片門檻。
  - 第 19 關暫定採用：平民 x8 + 猛男 x5 + 跑者 x4 + 盾兵 x3 + 工兵 x1。
  - 以上目標後續被修正：2 人裸裝不應以通過第 19 關為目標。
- 第三輪結果（加入建造範圍粗審計）：
  - 第三輪使用了錯誤假設，已作廢：它把建築外擴當成怪物更早能攻擊核心。
  - 所有怪物 x2 後，2 人裸裝約第 16 關死亡，太硬。
  - 若玩家太早把地基拉滿 35 格，Hitbox 外擴會讓核心更早受傷。
  - 依照新的裸裝定位，所有怪物 x2 可能反而接近正確壓力：2 人裸裝剛好過第 10，約第 15-16 關到極限，不應接近第 20。
  - 「平民/猛男 x2，跑者/盾兵/工兵先不 x2」會讓裸裝可過第 19，已判定太鬆，不作為目前目標。
- 第四輪結果（修正建築長度模型）：
  - 正確邏輯：建築長度是怪物接觸核心前的預射距離，不是更早受擊風險。
  - 若第 10 關前能做出 35 格有效建築長度，2 人裸裝可過第 19，死於第 20，明顯太鬆。
  - 若第 10 關有效建築長度只有約 20 格，2 人裸裝會死於第 10，太硬。
  - 下一輪重點應改為「資源可轉換成多少有效建築長度」，找出第 10-15 合理長度範圍。
- 第五輪結果（2x2 核心 + 神級建築土方成本）：
  - 核心改為 2x2 後，從底層內側往外最多 34 格。
  - 若要蓋 34 格神級玩法，成本改為左 34 + 右 34 + 高 68 = 136 土。
  - 左右平均挖礦時，2 人每關約 5.7 土，第 10 關約 57 土，只能支撐約 14 格，實測第 9 關死亡。
  - 修正後左右平均第 10 關只能支撐約 14 格，第 9 關死亡。
  - 極限優先右礦山拿土時，2 人每關約 9.12 土，第 10 關約 91 土，可支撐約 22 格，但仍於第 10 關死亡。
  - 136 土版本可防止神級建築過早成形，但目前對裸裝 2 人過第 10 偏嚴格。
- 第六輪結果（50 / 100 抽 + 第 10 關後卡片）：
  - 50 / 100 抽都能穩定通過第 15 關。
  - 但 50 抽已能突破第 20 關，依路線約死在第 23 或第 30。
  - 100 抽更強，依路線約死在第 28 或第 30。
  - 目前不符合「15 通關，但不超過 20」目標。
  - 問題主要來自裝備提高挖礦量後，同時放大土方、攻擊、攻速、射程與防禦成長。
  - 下一輪應優先測第 20 關 Boss 壓力、16-20 小怪曲線，或限制裝備對單局建築成長的放大。
- 第八輪結果（分段建築邊界）：
  - 0-10 關 10 格、11-20 關 20 格、21-30 關 30 格、30+ 關 35 格的方向可以壓住 100 抽不超過 20。
  - 100 抽左右平均路線可過 15，死於 20，接近目標。
  - 但 50 抽會死於第 10 關，拿不到卡片，第一段 10 格上限偏嚴格。
  - 下一輪建議測 0-10 關上限 12 / 14 格，或降低第 10 關 Boss 壓力。
- 第九輪結果（分段建築邊界 14 / 20 / 27 / 35）：
  - 50 抽左右平均路線可過第 15，約死於 16-17，符合目標。
  - 100 抽左右平均路線可過第 15，死於第 20，符合目標。

  - 100 抽右礦山可過第 15，約死於 17，不接近 20。
  - 建築邊界可暫定為：0-10 關 14 格、11-20 關 20 格、21-30 關 27 格、30+ 關 35 格。
- 第十輪結果（3 人 / 4 人同條件）：
  - 3 人 50 抽左右平均可過 15，約死 18-19，符合目標。

  - 下一輪應針對 100 抽設計第 20 Boss 或 21-30 成長，而不是再縮短 0-20 建築邊界。
- 第十一輪結果（普通怪血量 / 攻擊成長候選）：
  - Boss 更新候選：身高 4 格、攻擊距離 2 格。
  - 普通怪血量每關 +5% 並加上每 5 關攻擊 +1 時，全部路線過硬。
  - 單人 50 抽只能到第 6-7，100 抽也只到第 8-9，不符合目標。
  - 普通怪攻擊力「每 5 關 +1」不適合 1-20 關，會太早破壞 50 抽成長線。
  - 血量成長可保留候選，但建議改測 2-3%、或 5% 從第 11 關後開始；攻擊成長建議延後到第 20 關後。
- 第十二輪結果（A / B / C / A+D / B+D）：
  - 最值得討論的是 C：普通怪血量從第 11 關後開始每關 +5%，普通怪攻擊 1-20 不成長。

  - 鐵牙核心 +2 可控，沒有破第 20。
  - 厚土外殼 +25 / 回 25 可控，沒有破第 20。
  - 金輪核心 +0.25 會讓 2/3/4 人 100 抽越過第 20，過強。


## Must Solve 6：多人合作分工

- [x] 補出多人模式中 1-4 人的分工誘因與合作壓力。（MVP 版完成：不強制分工、不額外懲罰，提供多人意圖 Emoji 讓玩家自行合作）

### 目前問題

我們現在要解的是：Must Solve 6：多人合作分工。

核心問題不是「要不要做職業」，而是：

```md
多人模式如何在不強制分工、不額外懲罰玩家的前提下，提供足夠清楚的合作訊號。
```

現在卡住的是：

- 隊友不知道彼此現在想做什麼。
- 大家可能同時去挖礦，家裡沒人修；這應該是自然風險，不做額外系統懲罰。
- 大家可能同時守家，資源不夠；這也應該由玩家自行判斷。
- 有人想建築或搬運，但隊友看不出來。
- 有人按危險警告時，隊友需要快速知道「回家守核心」。

### 我的建議

先做一個 MVP 版「多人意圖標記系統」。

這不是聊天系統，也不是職業鎖定，而是讓隊友快速知道：

```md
我現在要挖礦。
我現在要建築。
我現在要搬運。
我現在要修復。
家裡危險，請回防。
```

### 多人意圖 Emoji

| Emoji | 意圖 | 用途 |
|---|---|---|
| ⛏️ | 挖礦 | 我要去挖礦 / 我負責資源 |
| 🧱 | 建築 | 我要蓋塔 / 補地基 |
| 🦵 | 搬運 | 我要跑腿，把資源帶回家 |
| 🔧 | 修復 | 我要守家修塔 |
| ⚠️ | 危險 / 回防 | 家裡危險，呼喚隊友回家守核心 |

### 顯示位置

Emoji 同時顯示在兩個地方：

```md
1. 正上方房員欄：玩家頭像旁邊顯示 30 秒。
2. 角色頭上：浮出該 Emoji，顯示 30 秒。
```

玩家可以隨時更換意圖。

白天 / 黑夜切換時是否保留：

```md
待定。
```

目前不強制清空，先保留作為待實測項目。

### 操作方式

桌面版：

```md
按住 Alt 開啟意圖輪盤。
滑鼠往上 / 下 / 左 / 右選擇。
操作概念類似 LOL Emoji / ping wheel。
```

手機版：

```md
使用 Emoji 按鈕。
點開後選擇 ⛏️ / 🧱 / 🦵 / 🔧 / ⚠️。
```

### ⚠️ 危險 Emoji 規則

⚠️ 不是普通表情，而是回防訊號。

用途：

```md
核心正在掉血。
Boss / 加時壓力太高。
家裡沒人修。
需要隊友回家提供靈力與修復。
```

設計重點：

- 守家玩家可以用 ⚠️ 叫隊友回來。
- 回家玩家可以提供靈力，讓核心攻擊 / 攻速更高。
- 這讓「守家」不只是修血，也包含呼叫隊伍協作。

### 能不能用模擬解決？

部分可以。

可以模擬：

- 如果 1 人守家、其他人外出，核心 DPS 下降多少。
- ⚠️ 後隊友回家需要幾秒，是否來得及救核心。
- 多人都離家時，核心危險警告觸發頻率。

不適合只靠模擬：

- 玩家是否真的會看 Emoji。
- Alt 輪盤是否順手。
- 手機 Emoji 選單是否太慢。
- 隊友會不會自然配合 ⚠️ 回防。

這些需要 MVP 實玩驗證。

