## 4. 與 Codex 的分工模式（重要）

**Codex 負責**：遊戲性數值/平衡相關的一切——`mustsolve.md` 的7個核心設計問題（怪物職能設計、建築策略、難度曲線、卡片系統、多人合作分工、MVP驗證標準）、`simulation/` 的模擬驗證。這些題目都需要跑模擬、用 Top-down 方式反推數值。

**Claude Code（你）負責**：技術架構審查面——系統會不會在真實情況下出問題，而不是數值好不好玩。目前已識別出4大類缺口（細節見第6節）：
- 安全性／反作弊（輸入驗證、Rate limiting）
- 多人連線健壯性（斷線重連、PeerJS拓撲——拓撲已解決）
- 資料可維護性（存檔版本相容性）
- 營運缺口（金流串接、隱私合規、新手教學）

**跨領域協作流程**（已建立的固定模式）：
1. 當你提出一個自己無法完全解決、或需要跨領域意見（例如牽涉到Codex正在跑的模擬/數值）的架構問題時，**主動寫一份簡潔、自成一體的「核心問題+技術選項對照」簡報**（問題是什麼、有哪些選項/trade-off、你的建議），讓使用者可以直接複製貼給Codex討論，不需要使用者自己整理
2. 使用者跟Codex討論完，會把Codex的結論帶回來給你
3. **一旦某個架構/設計決策同時經過Codex和你審核都同意，就可以直接同步進正式文件，不需要再額外跟使用者確認一輪**——這是已經建立的標準，不用每次都多問一次「要不要寫進文件」

**討論格式**（兩邊都遵循類似結構，但用詞依領域微調）：
```

[建議方案 + 理由]

## 能不能用模擬解決？（Codex的數值類問題）

## 能不能用其他技術解決？（架構類問題）

```

## 5. 已確定的技術架構關鍵事實（避免重新推導，但請以實際文件為準）

- **技術棧**：HTML+JS（ES Module）/ PeerJS（多人）/ Supabase（帳號+存檔）/ Vercel（部署）
- **Multiplayer 拓撲**：Star／房主中心（非Full Mesh）。非房主玩家只跟房主建1條WebRTC連線，玩家操作送房主→房主驗證/處理→房主廣播結果。這個決定已經過Codex確認並同步進`game-architecture-plan.md`
- **房主權威模型**：怪物AI、傷害計算都在房主端判定；存檔發起也只有房主能做
- **Event Log**：房主端統一批次上傳（每30秒），每筆事件附帶 `player_uid` 標記是哪位玩家操作，不影響RLS設計
- **Save File 是5層結構**（地圖層/進度層/玩家層/共享資源層/房間層），跟「玩家帳號資料」是分開的兩套資料（帳號資料跨房間通用，Save File只跟著單一房間）
- **雲端基礎設施**（詳見`cloud-infra-cost-plan.md`）：因為多人同步走PeerJS P2P不是Supabase Realtime，Supabase負擔主要是帳號驗證+存檔讀寫+批次上傳，不是典型「同時連線數」算法。PeerJS訊號伺服器建議自架（不用官方免費Cloud Server），TURN建議先用Open Relay Project免費額度
